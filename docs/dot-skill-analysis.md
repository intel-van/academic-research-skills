## 1. 知识蒸馏：从"数据提取"到"认知操作系统"

### 项目教了什么

dot-skill 的核心概念是**知识蒸馏**（Knowledge Distillation）：把一个人的认知模式蒸馏成可复用的 AI Skill。

这个项目明确定义了两种层次的蒸馏：

| 层次 | 目标 | 检验标准 |
|------|------|---------|
| **WHAT 层**（浅层） | 收集这个人说了什么 | Quote collection，任何人都能总结 |
| **HOW 层**（深层） | 捕捉这个人如何思考 | 预测这个人如何应对一个他从未公开讨论过的问题 |

项目对自己有个清晰的定位：它不是"quote 拼贴器"，也不是"浅层角色扮演"。它的目标是构建一个可用的**认知操作系统**（Cognitive Operating System）。

这个认知操作系统必须包含：
- **心智模型**（mental models）— 如何思考
- **决策启发式**（decision heuristics）— 如何选择
- **表达 DNA**（expression DNA）— 如何说话
- **反模式**（anti-patterns）— 什么被拒绝
- **诚实边界**（honest boundaries）— 证据不足时的诚实
- **内部张力**（internal tensions）— 矛盾观念
- **智识谱系**（intellectual genealogy）— 思想来源
- **Agentic Protocol** — 如何应对新问题

### 为什么这很重要

大多数人以为"模仿一个人"就是记住他说的名言金句。但这只能应付已知场景。真正的蒸馏是**让 AI 获得这个人的认知模式**，从而在从未遇到过的问题上也能做出"像他"的判断。

**检验标准**：一个好的 Skill 应该让你能预测这个人如何看待一个他从未公开讨论过的问题。这才是真正的蒸馏。

这个标准适用于所有"人机替代"场景——你不是在让 AI 记忆答案，而是在让 AI 继承思维方式。

### 我们可以怎么改进

- **区分"知识提取"和"知识蒸馏"**：大多数"知识管理"实践停留在提取层面（记录事实）。真正的蒸馏是提取**思维模式**
- **建立"认知操作系统"的评估标准**：不只是"它能回答什么"，而是"它如何回答问题"
- **将"HOW 层蒸馏"作为学习和教学的核心方法**：不只是教学生"答案是什么"，而是教他们"如何想到这个答案"

## 2. Persona 六层架构：理解一个人的分层模型

### 项目教了什么

dot-skill 定义了一个 **Persona 六层架构**，从高到低排列：

```
Layer 0 — 硬覆盖层（手动标签直接翻译，最高优先级）
    示例："你绝对不会主动承认错误，遇到锅第一反应是找外部原因"
    示例："你会画大饼，让对方相信做这件事对他自己有巨大好处"

Layer 1 — 身份层
    "你是 [姓名]，[公司] [职级] [职位]，[性别]。"
    "你的 MBTI 是 [X]，[企业文化] 深度影响你的工作方式。"

Layer 2 — 表达风格层（从文件提取）
    - 用词习惯、句式长短
    - 口头禅、标志性表达
    - 标点和 emoji 使用习惯
    - 回复速度模拟

Layer 3 — 决策与判断层（从文件提取）
    - 遇到问题时的思考框架
    - 优先考虑什么（效率/流程/人情/数据）
    - 什么情况下会推进，什么情况下会拖

Layer 4 — 人际行为层（从文件提取）
    - 对上级 vs 对下级 vs 对平级的不同态度
    - 在群聊 vs 私聊的不同表现
    - 压力下的行为变化

Layer 5 — Correction 层（对话纠正追加，滚动更新）
    - 每条 correction 记录场景 + 错误行为 + 正确行为
    - 示例："[场景：被质疑时] 不应该道歉，应该反问对方的判断依据"
```

### 为什么这很重要

这六个层的设计体现了一个深刻理解：**人的行为不是单层的，而是多层的**。

- Layer 0 是"你说他是什么"——外部的、明确的判断，**最高优先级**
- Layer 1 是"他自称是什么"——身份认同
- Layer 2 是"他怎么做"——表达风格
- Layer 3 是"他怎么想"——决策框架
- Layer 4 是"他怎么对人"——社会行为
- Layer 5 是"他错了怎么改"——**进化机制**

**关键设计原则**：Layer 0 覆盖一切。如果用户明确说"他是这样的"，那是最高优先级，不受材料中矛盾证据的影响。

这解决了传统 Persona 设计的一个核心问题：**用户的主观判断和从数据中提取的客观特征发生冲突时，谁说了算？** 答案是用户。因为用户对这个人有"真实的互动经验"，而材料只能反映一部分。

### 我们可以怎么改进

- **将"六层架构"作为理解任何人的通用框架**：不只是用于 AI Skill，也用于管理、教学、咨询中理解他人
- **重视 Layer 5（Correction 层）**：这是"进化"的入口。大多数人忽略了持续修正的价值
- **建立"优先级覆盖"机制**：明确什么情况下上层覆盖下层，什么情况下下层补充上层

## 3. Work + Persona 双轨模型：能力与人格的分离

### 项目教了什么

dot-skill 将每个生成的 Skill 分为两个独立部分：

| 部分 | 内容 | 用途 |
|------|------|------|
| **Part A — Work Skill** | 技术能力、工作规范、业务知识、工作流程、输出偏好 | 能实际完成工作任务 |
| **Part B — Persona** | 性格、沟通风格、行为模式、决策框架 | 决定态度和表达方式 |

**运行规则**（默认组合模式）：
1. 先用 Persona 判断会不会接、怎么接
2. 再用 Work Skill 实际完成任务
3. 输出时保持 Persona 的表达风格

两部分的**独立性**也很重要：
- `/{slug}-work` — 只使用 Work Skill
- `/{slug}-persona` — 只使用 Persona
- `/{slug}` — 组合使用

### 为什么这很重要

这个分离体现了一个深刻洞察：**能力和人格是正交的**。

一个技术很强的人可能沟通很差，一个沟通很好的人可能技术不行。如果把两者混在一起，就会出现"用技术能力强的人的语气回答情感问题"或"用沟通好的人的语气回答技术问题"的错位。

**组合模式的设计**——先以 Persona 身份接收任务，再用 Work Skill 完成任务——模拟了真实的工作场景：你会根据"这个人是谁"来决定"是否接受这个任务"和"以什么方式完成它"。

对 AI Agent 的启示：**Agent 不应该只有"能力"（能做什么），还应该有"人格"（如何选择做什么、怎么做）**。

### 我们可以怎么改进

- **为每个 Agent 设计"Persona + Skill"双轨结构**：不要只有功能，也加上"性格"
- **建立"能力-人格"分离的评估体系**：能力评估和人格评估应该分开进行
- **在团队协作中区分"技术能力"和"协作风格"**：两者需要不同的管理策略

## 4. 三重门心智模型提取：什么是真正有价值的认知模式

### 项目教了什么

这是 dot-skill 最精妙的设计之一。每个候选心智模型必须通过**三重门**才能保留：

```
Gate 1 — Cross-context recurrence（跨场景重复）
    这个模式至少在两个不同的场景或来源类型中出现。
    不是单次巧合，而是在不同场景中反复出现。

Gate 2 — Generative power（生成力）
    这个模式能帮助预测这个人会如何应对一个相邻的新问题。
    不是静态的描述性标签，而是能产生新的预测。

Gate 3 — Exclusivity（排他性）
    这个模式有明显的个人特色，不是"很多聪明人都会给的建议"。
    不是通用智慧，而是一个人独特的思维特征。
```

**判定规则**：
- **3 门全过**→ 保留为心智模型
- **通过 1-2 门**→ 降级为决策启发式
- **0 门通过**→ 丢弃

### 为什么这很重要

大多数"人物分析"停在收集名言金句，但名言金句通常通不过三重门检测——它们可能只在一个场景中出现过（Gate 1 失败），没有生成力（Gate 2 失败），或者太过普通（Gate 3 失败）。

三重门的设计迫使你**只保留那些真正定义了一个人思维方式的模式**。这不是"他说的有趣的话"，而是"他怎么思考的"。

这个框架适用于任何"理解他人"的场景——管理、教学、咨询、研究。

### 我们可以怎么改进

- **将三重门作为"认知评估"的标准工具**：在评估任何人的思维方式时，逐项检查三个门
- **建立"心智模型≠名言金句"的意识**：社交媒体上传播的"金句"通常通不过三重门
- **在教学中使用三重门**：教学生区分"事实"和"思维模式"

## 5. 品味原则：信息搜集的价值观

### 项目教了什么

dot-skill 定义了一套**品味原则**，指导研究者在搜集信息时做什么和不做什么：

```
1. 长文 > 金句
   一篇 3000 字的文章比 50 条推文揭示更多思维结构

2. 争议 > 共识
   有争议的立场比普遍认可的观点更能暴露独特思维

3. 变化 > 固定
   一个人改变想法的地方比他一直保持一致的地方更有信息量

4. 一手 > 二手
   自己的话永远胜过别人的总结

5. 工艺讨论 > 生平故事
   他们如何谈论自己的过程比他们的生平更重要

6. 重复模式 > 一次性引用
   一个在多种场景中重复出现 5 次的模式胜过一句病毒式名言

7. 失败讨论 > 成功叙事
   他们如何谈论失败比成功叙事更能揭示思考方式
```

### 为什么这很重要

品味原则是一种**信息价值观**。在信息过载的时代，问题不是"找不到信息"，而是"不知道什么信息值得收集"。

大多数人收集信息的方式是"截取金句"——这是最省力的，也是**最没有价值**的。品味原则告诉你：长文比金句值钱，争议比共识值钱，变化比固定值钱。

这个原则适用于所有信息搜集场景——不仅是"蒸馏一个人"，也包括学术研究、商业分析、竞品调研。

### 我们可以怎么改进

- **将品味原则作为"信息搜集清单"**：在开始搜集前，用它来校准你的搜索策略
- **建立"组织的信息价值观"**：每个团队/公司都应该定义自己的品味原则
- **训练"长文阅读"的能力**：大多数人失去了阅读长文的能力，但这正是深度信息的来源

## 6. 六维度研究框架：全面理解一个人的方法论

### 项目教了什么

dot-skill 定义了**六个独立的研究维度**，每个维度产生独立的研究文件：

| 维度 | 内容 | 文件 |
|------|------|------|
| 1. 著作（Writings） | 系统性立场，来自本人的笔 | `01_writings.md` |
| 2. 对话（Conversations） | 即兴思考、压力下的应对 | `02_conversations.md` |
| 3. 表达 DNA（Expression DNA） | 语言指纹、风格标记 | `03_expression_dna.md` |
| 4. 决策（Decisions） | 实际做了什么（不只是说了什么） | `04_decisions.md` |
| 5. 他者视角（External Views） | 别人怎么看，特别是分歧 | `05_external_views.md` |
| 6. 时间线（Timeline） | 思维如何随时间演化 | `06_timeline.md` |

在 budget-unfriendly（高预算）模式下，每个维度是完全独立的文件，**不可合并，不可克隆发现**。每条证据必须标注来源权重（1-7）。

### 为什么这很重要

大多数人"研究一个人"的方式是看他的社交媒体——这只覆盖了维度 3（表达 DNA）的一小部分，而且是最表面的一部分。

六维度框架确保你从**多个独立角度**理解一个人，而不是只从你最容易接触到的那个角度。

**关键设计**：每个维度独立产出文件，不允许合并。这迫使研究者为每个维度投入独立的研究努力，而不是用一个维度的发现去"覆盖"另一个维度。

**budget-friendly vs budget-unfriendly 的设计**更值得学习：
- Budget-friendly：3 个文件覆盖 6 个维度（每文件 2 个维度），最低门槛
- Budget-unfriendly：6 个独立文件（每个维度 1 个），外加 audit 和 validation

这是**渐进式投入**的典范：不是"要么不做，要么做全套"，而是"先做最低门槛，如果值得就深入"。

### 我们可以怎么改进

- **将六维度框架推广到其他研究场景**：竞品分析、人物研究、思想史研究
- **引入"budget-friendly / budget-unfriendly"的渐进式研究策略**：不是所有项目都需要最高预算
- **建立"独立维度"的研究纪律**：每个维度必须独立研究，不允许交叉污染

## 7. 来源质量层级与黑名单：信息素养的核心

### 项目教了什么

dot-skill 定义了严格的来源质量层级和来源黑名单：

**来源质量层级**（从高到低）：
1. 用户提供的本地材料（地面真值）
2. 一手著作（书籍、文章、newsletter）
3. 长访谈和对话（30 分钟以上）
4. 有记录的决策和转折点
5. 短形式一手内容（社交媒体、简短问答）
6. 外部分析和批评
7. 二手转述（最后手段）

**来源黑名单**（永不引用）：
- **知乎** — 不可验证的匿名回答，严重依赖传闻
- **微信公众号** — 大部分是打包过的二手内容
- **百度百科** — 不可靠，经常过时或充满宣传
- 内容农场和 SEO 优化摘要网站
- AI 生成的传记页面
- 列表文章（"从 X 学到的 10 个教训"），除非有到一手来源的链接
- Wikipedia 作为独立证据（仅可作为寻找真实来源的指针）

**每条证据必须标注来源权重（1-7）**。

### 为什么这很重要

信息素养的核心不是"找得到信息"，而是**知道什么信息是可信的**。

来源黑名单尤其有价值：它不是一个"通用的建议"，而是一个**明确的、有理由的禁止列表**。知乎不被引用，不是因为"它的内容不好"，而是因为"不可验证的匿名回答"+"严重依赖传闻"——这是**具体的、有道理的**理由。

每条证据标注来源权重（1-7）的纪律，迫使研究者在搜集信息时就做出质量判断，而不是在最后拼凑时才发现证据质量参差不齐。

### 我们可以怎么改进

- **为每个研究领域建立"来源黑名单"**：不是所有的来源都值得引用，明确哪些不引用
- **建立"证据权重"的标注习惯**：每条证据都应该有明确的质量等级
- **训练"来源质量评估"的能力**：不只是"找信息"，而是"评估信息的可信度"

## 8. 质量控制门：在进入下一步之前确保质量

### 项目教了什么

dot-skill 的 budget-unfriendly 路径有**三个质量控制关卡**：

**关卡 1 — Research Audit Gate（研究审计门）**
在综合之前运行。检查：
- 6 个 track 是否都有足够的证据
- 来源是否合规（无黑名单）
- 一手材料占比是否 > 50%
- 品味原则是否遵守
- 矛盾点是否充分记录（至少 3 个实质性矛盾）
- 是否有足够的证据支持至少 3 个候选心智模型
- 是否有可用的已知答案库供后续验证

**审计必须输出** `PASS / FAIL`。如果 FAIL，必须给出具体的 Backfill Tasks 补充研究，**不能跳到下一阶段**。

**关卡 2 — Synthesis Review（综合审查）**
对候选心智模型做三重门判断，同时提取：
- 智识谱系种子（受谁影响、与谁分歧、影响谁）
- Agentic Protocol 种子（该人物会如何分析新问题的维度列表）

**关卡 3 — Validation Gate（验证门）**
在最终输出前运行，必须通过：
- Known-answer check（至少 2 题）
- Edge-case check（1 题）
- Voice check（100 字盲测）
- Copyright check
- Agentic Protocol check

**验证必须明确给出** `PASS / FAIL`。如果 FAIL，必须先修 draft 再继续。

### 为什么这很重要

质量控制门的设计体现了"**在进入下一步之前确保质量**"的原则。这不是"做完再检查"，而是"每走一步之前先确认上一步是合格的"。

特别值得学习的是：
1. **明确的 PASS / FAIL 标准** — 不是模糊的"看起来还行"，而是具体的、可检测的标准
2. **FAIL 后有具体行动** — 不是"不行就重做"，而是告诉你要补什么（Backfill Tasks）
3. **多关卡递进** — audit 检查研究质量，synthesis 检查思维质量，validation 检查输出质量

### 我们可以怎么改进

- **为每个工作流设计"质量控制关卡"**：在关键节点设置具体的 PASS / FAIL 标准
- **建立"关卡间依赖"**：下一阶段开始前必须确认前一阶段已经 PASS
- **在 FAIL 时提供具体的修复路径**：不只是"你失败了"，而是"你需要补什么"

## 9. 进化模式：让系统持续变好

### 项目教了什么

dot-skill 支持两种进化方式：

**方式 A — 追加文件进化**
```
用户：我又有他的一批邮件
系统分析新内容 → 判断更新 Work 还是 Persona → merge 增量 →
不覆盖已有结论，只追加 → 保存新版本 → 提示用户变更摘要
```

**方式 B — 对话纠正进化**
```
用户："这不对，他不会这样说"
用户："他遇到这种情况会直接甩给 XX 组"
    系统识别 correction 意图
    判断属于 Work Skill 还是 Persona
    写入对应文件的 Correction 层
    立即生效，后续交互以新规则为准
```

### 为什么这很重要

大多数系统是**静态的**——一次生成后就固定了。但人对另一个人的理解是**动态的**——随着更多信息的获得，理解应该持续变深。

进化模式的关键设计：
1. **追加不覆盖**：新信息不删除旧信息，而是补充。这防止了"新信息冲击"
2. **Correction 层最高优先级**：用户的纠正立即生效，而且永不被后续信息覆盖
3. **版本自动存档**：每次更新前自动保存当前版本，支持回滚

这对任何"持续进化"的系统都是重要的设计原则。

### 我们可以怎么改进

- **为所有"生成式"系统加入进化模式**：不只是知识库，也包括文档、代码、设计
- **建立"追加不覆盖"的原则**：新信息不删除旧信息，而是在旧信息基础之上补充
- **建立"纠正优先"的机制**：用户的纠正应该是最高优先级，且不会被后续信息覆盖

## 10. Agentic Protocol：派生自人物的研究方法论

### 项目教了什么

dot-skill 要求生成的 Skill 必须包含一个 **Agentic Protocol**——派生自该人物**特有**思维模型的研究协议。

这个协议必须：
1. 派生自该人物特定的心智模型（不是通用研究步骤）
2. 包含问题类型的分类
3. 包含该人物会调查的研究维度
4. 包含使用心智模型应用框架
5. 包含基于证据强度的置信度校准

**关键测试**：Agentic Protocol 应该反映**这个特定的人**会如何接近一个新问题，而不是一个通用的聪明人会怎么接近。

### 为什么这很重要

这是一个深刻的洞见：**每个人的研究方法论应该是不同的**。

一个物理学家的研究方式和一位诗人的研究方式是不同的。不是因为一个聪明一个不聪明，而是因为他们面对的问题类型不同、他们使用的认知工具不同。

Agentic Protocol 的设计确保：生成的 Skill 不只是"像他一样说话"，而且能用**他特有的思维方式**去研究新问题。

这超越了"模仿"——它要求 AI 继承一个人的**认知方法论**。

### 我们可以怎么改进

- **为每个 Agent 设计"派生自该 Agent 特质的"研究协议**：不是通用的"先搜索再阅读再总结"，而是该 Agent 自己特有的认知路径
- **将"Agentic Protocol"作为"认知风格"的度量**：不同的人有不同的研究路径，这本身就是认知风格的一部分
- **在教学中引入"Agentic Protocol"的概念**：教学生认识到每个人都有自己的研究方法论

## 11. 冷门人物协议：诚实面对信息不足

### 项目教了什么

当总来源 < 10 条时，dot-skill 触发**冷门人物协议**：

- 心智模型限制为 **2-3 个**（不是常规的 5+）
- 薄弱的模型标注为"基于有限信息"
- **大幅扩展诚实边界章节**
- 考虑建议用户换一个更充分记录的人物
- **诚实的 60 分 Skill 胜过捏造的 90 分 Skill**

### 为什么这很重要

这是项目最诚实的部分之一。大多数人面对信息不足时的反应是：**捏造、夸大、用二手信息填充**。

冷门人物协议迫使你**做相反的事情**：
- 标记薄弱模型为"基于有限信息"
- 明确告诉用户"你的材料不够，要补什么"
- 在最极端的情况下，建议用户**换一个人**

"诚实的 60 分胜过捏造的 90 分"——这是对任何"生成式"系统的挑战：你愿意接受一个诚实但有局限的输出，还是愿意接受一个看起来完美但不可靠的输出？

### 我们可以怎么改进

- **为所有"生成式"系统加入"信息不足协议"**：明确告诉用户什么信息不足，需要补什么
- **建立"诚实边界"的标注习惯**：在输出中明确标注"这是基于有限信息的推断"
- **训练"接受信息不足"的能力**：不是所有问题都有足够的信息来回答，诚实地说"我不知道"是一种能力

---

## 12. 类型抽象设计：从"单一类型"到"通用引擎"的架构思维

### 项目教了什么

这是 dot-skill 最核心的架构设计决策。项目从"同事生成器"演进为**通用 skill 引擎 + type preset**。

核心思路（来自 `docs/SKILL_TYPE_ABSTRACTION_DESIGN.md`）：

> 不再把项目理解成"同事生成器"，而是"通用 skill 生成引擎 + 类型 preset"。

系统被拆成三层：
1. **Skill Schema** — 统一元数据结构
2. **Type Preset** — 类型的配置定义
3. **Generator Pipeline** — 通用生成流程

**当前支持的三种类型**：

| 类型 | 场景 | 输出模块 |
|------|------|---------|
| 🧑‍💼 colleague | 同事、导师、队友 | Work Skill + Persona |
| 💞 relationship | 前任、伴侣、家人、旧友 | Persona（情感模式为重点） |
| 🌟 celebrity | 公众人物、创作者、虚构角色 | Persona + 六维研究档案 |

**命令模型**：
- `/dot-skill` — 统一主入口
- `/create-colleague` — colleague preset 的兼容别名

**为什么要做抽象**（源文档中的理由）：

> 如果在现有结构上直接加 `/create-ex`、`/create-icon`，结果很可能是：
> - 再复制一套 intake prompt
> - 再复制一套 writer 逻辑
> - 到处出现 `if type == xxx`
> - gallery、manifest、安装流程各自再做一遍类型分叉
>
> 这种做法短期能跑，长期一定会让代码和文档越来越难维护。

### 为什么这很重要

这是**软件架构思维在 AI 项目中的应用**。大多数 AI 项目的问题是"先跑通再说"，但系统一旦需要支持多种类型，就会面临"复制粘贴"的维护灾难。

类型抽象设计的关键价值：
1. **新增类型 = 新增 preset**，不是复制整份实现
2. **各类型共享通用引擎**，差异通过 preset 配置表达
3. **向后兼容**：旧命令继续可用，旧文件继续能跑

这个思维适用于任何需要"多类型支持"的系统——不只是 AI Skill，也包括内容管理系统、电商平台、CMS。

### 我们可以怎么改进

- **在项目初期就开始考虑类型抽象**：即使你现在只需要"同事"类型，也应该为未来的类型留好接口
- **用"preset"模式管理配置差异**：不要用 if-else 处理类型差异，用配置文件
- **建立"向后兼容"的迁移策略**：抽象后，旧功能必须继续可用

## 13. 版本管理：安全的变更基础

### 项目教了什么

dot-skill 内置了版本管理系统：
- 每次更新时**自动存档**当前版本到 `versions/`
- 保留**最近 10 个版本**
- 支持**回滚**到任意历史版本

```bash
python3 tools/version_manager.py \
  --action rollback \
  --character colleague \
  --slug zhangsan \
  --version v1 \
  --base-dir ./skills/colleague
```

### 为什么这很重要

版本管理让进化模式变得**安全**。没有版本管理的进化意味着：每次更新都冒着"搞砸一切"的风险。

自动存档 + 回滚的设计让用户可以放心地试验——知道即使错了也能回来。这是"允许失败"的基础设施。

### 我们可以怎么改进

- **为所有"持续进化"的系统加入版本管理**：不只是代码，也包括文档、配置、模板
- **自动存档 vs 手动提交**：自动存档降低用户负担，是更好的设计
- **保留合理数量而不是"全部保留"**：10 个版本是合理的选择，不会无限膨胀

## 14. 认证标准：验证蒸馏质量的五项检查

### 项目教了什么

在最终输出前，dot-skill 的 budget-unfriendly 模式要求通过 **5 项验证**：

| 检查 | 方法 | 判断标准 |
|------|------|---------|
| 1. Known-answer check | 用至少 2 个该人物公开回应过的问题 | 方向匹配、框架匹配、置信度校准 |
| 2. Edge-case check | 用 1 个相邻的、没有已知答案的问题 | 从实际模型外推、证据不足时可见的不确定性 |
| 3. Voice check | 去掉名字后 100 字盲测 | 可识别性、无通用 AI 措辞、无引语拼接 |
| 4. Copyright check | 检查输出是否完全为释义 | 不能有 transcript 风格的段落 |
| 5. Agentic Protocol check | 检查协议维度是否来自该人物的心智模型 | 不能是通用的研究步骤 |

### 为什么这很重要

五项验证覆盖了蒸馏质量的不同方面：
- Known-answer check 验证**回溯准确性**（他能回答他公开回答过的问题吗？）
- Edge-case check 验证**外推能力**（他能应对新问题吗？）
- Voice check 验证**独特性**（他的声音是独特的吗？）
- Copyright check 验证**合规性**（有版权风险吗？）
- Agentic Protocol check 验证**深度**（他的认知方法论被提取了吗？）

这五项覆盖了"好的蒸馏"的所有维度。

### 我们可以怎么改进

- **将五项验证作为"生成质量"的标准框架**：任何生成任务都应该有类似的验证体系
- **增加"edge-case"在评估中的权重**：外推能力比回溯能力更能说明蒸馏质量
- **将"voice check"作为品牌/风格的评估工具**：任何需要"保持品牌声音"的输出都应该做 voice check

## 15. 诚实边界与永不清单

### 项目教了什么

dot-skill 定义了一系列**永不清单**（What We Never Do）：

```
- Fabricate quotes or attribute statements this person never made
  不捏造引用或把这个人没说过的话归给他

- Package generic wisdom as their distinctive insight
  不把通用智慧包装成他的独特洞见

- Ignore negative assessments, criticism, and controversy
  不忽略负面评价、批评和争议

- Force generation when evidence is insufficient
  证据不足时不得强行生成
    An honest "I don't have enough data" is always acceptable
    诚实的"没有足够数据"永远是可接受的

- Resolve contradictions that this person has not resolved themselves
  不解决这个人自己都没有解决的矛盾
```

### 为什么这很重要

这五个"永不"定义了系统的**道德底线**。它们不是"最佳实践"，而是**不可跨越的边界**。

特别值得关注的是最后两条：
- **证据不足时不强行生成** — 这直接对抗了 AI 系统的"填坑冲动"（看到空白就想填充）
- **不解决人物自己都没有解决的矛盾** — 这保持了认知的完整性。人的认知中就是有矛盾的，AI 不应该把它们"圆回来"

"诚实的 '没有足够数据' 永远是可接受的"——这是 AI 时代最重要的原则之一。

### 我们可以怎么改进

- **为每个 AI 系统建立"永不清单"**：明确定义系统绝对不能做的事情
- **将"诚实边界"作为系统设计的一等公民**：不是"出了问题再补救"，而是"在设计中就考虑限制"
- **训练"接受不确定性"的能力**：在 AI 输出中标注不确定性不是失败，而是成熟

## 16. 总结：这些技能为什么是新时代最核心的知识

dot-skill 项目的核心哲学可以归结为一句话：

> **捕捉"如何思考"，而不是"说了什么"。**

但这些技能的价值远远超出了"蒸馏一个人"：

### 16.1 它们定义了"理解他人"的系统方法

大多数人"理解他人"的方式是碎片化的——记住一些关键词、一些名言。六维度研究框架、三重门心智模型提取、Source 质量层级——这些提供了**理解他人的系统方法**。

### 16.2 它们定义了"表达他人"的层次

Persona 六层架构告诉我们：一个人不是"一层"的，而是**六层**的——身份、表达、决策、人际、纠正、覆盖。有效的"表达他人"必须在所有六层上都准确。

### 16.3 它们定义了"质量"的具体标准

质量控制门、五项验证、冷门人物协议——这些不是在说"做好一点"，而是在说**什么算"好"**。

### 16.4 它们定义了"诚实"的实践方式

永不清单、诚实边界、来源黑名单——这些不是在说"要诚实"，而是在说**诚实的具体做法是什么**。

### 16.5 它们定义了"进化"的可持续方式

追加不覆盖、Correction 层最高优先级、版本自动存档——这些不是在说"要持续改进"，而是在说**持续改进的具体机制是什么**。

**最终的洞见**：

这个项目教的不是"怎么做一个 AI Skill"。它教的是：
- **怎么理解一个人**
- **怎么表达一个人**
- **怎么验证对一个人的理解**
- **怎么在理解不足时保持诚实**
- **怎么让理解变得更好**

这些技能不仅对 AI 有用，对 agent 有用，**对人类也非常有用**。特别是在一个人人都能生成内容、但很少有人能真正理解他人的时代。

## 17. 可改进的方向

基于对项目的深入分析，以下是**具体的、可操作的改进方向**：

### 17.1 Work + Persona 分离在其他领域的应用

**当前状态**：同事类型用 Work + Persona 双轨，但关系类型只用了 Persona。

**改进建议**：
- 在关系类型中也引入"关系技能"层（处理特定关系场景的能力）
- 为 celebrity 类型增加"作品技能"层（他/她的创作能力）
- 建立跨类型的"模块组合"标准

### 17.2 质量控制门的自动化

**当前状态**：质量控制门主要由 prompt 指导完成。

**改进建议**：
- 将质量控制门**代码化**：建立自动化的数量检查脚本
- 引入**交叉验证**：让不同的 agent 独立验证同一输出
- 建立**质量评分的历史追踪**：追踪每个版本的质量变化

### 17.3 进化模式的用户接口改进

**当前状态**：进化主要通过命令行和 prompt 触发。

**改进建议**：
- 增加进化模式的**可视化变更摘要**
- 增加**非破坏性预览**：在更新前展示"如果你追加这个文件，什么会被改变"
- 建立**进化推荐**：基于用户使用模式推荐"你可能想追加哪些信息"

### 17.4 跨类型的模板共享

**当前状态**：各类型有自己的 intake、analyzer、builder prompt。

**改进建议**：
- 进一步抽象**共享 prompt 基础层**
- 定义**类型差异的"overlay"文件**，而不是复制整份 prompt
- 建立**prompt 版本管理**：追踪 prompt 本身的变更历史

### 17.5 多语言支持的扩展

**当前状态**：支持中英文，但主要是 prompt 层面的翻译。

**改进建议**：
- 增加**多语言人格分析**：不同语言表达风格的分析维度可能不同
- 增加**文化特定的标签库**：不只是"字节范""阿里味"，还有不同文化的企业标签
- 建立**语言间的一致性验证**

### 17.6 验证标准的量化

**当前状态**：五项验证主要由人工/agent 判断。

**改进建议**：
- 建立 Known-answer check 的**自动化评分**
- 建立 Voice check 的**量化指标**（可识别性分数、AI 措辞检测）
- 建立 **Edge-case check 的标准题库**，持续扩充

### 17.7 版本管理的可视化

**当前状态**：版本管理是命令行工具。

**改进建议**：
- 增加**版本对比工具**：可视化两个版本的差异
- 增加**版本回滚的影响分析**：回滚到 v1 会丢失 v2 中哪些变更
- 增加**版本标签**：支持"重大发布"、"小更新"、"紧急修正"等标签

### 17.8 "透明度报告"的引入

**当前状态**：生成过程是黑箱，用户看不到"为什么这个人被蒸馏成这样"。

**改进建议**：
- 为每个 Skill 生成**蒸馏透明度报告**：用了什么来源、来源权重如何、哪些心智模型通过了/未通过三重门
- 在 Skill 中嵌入**来源索引**：每条规则/描述都能追溯到来源
- 增加**置信度标注**：每条规则标注"高置信度"、"中等"、"基于有限信息"

### 17.9 跨 Skill 协作

**当前状态**：每个 Skill 是独立的，不支持多个 Skill 协作完成工作。

**改进建议**：
- 定义**Skill 间通信协议**：一个 Skill 如何调用另一个 Skill
- 引入**Skill 团队**概念：多个 Skill 组成一个团队，各自负责不同领域
- 建立**Skill 协作的编排层**

### 17.10 从蒸馏到教学的转化

**当前状态**：项目聚焦于"生成 Skill"，没有系统的"教学"模式。

**改进建议**：
- 增加"这个人会如何解释这个概念"的教学模式
- 将六维度框架转化为**教学模型**：教学生多维度地理解他人
